# Sol GPT 家族编排

## 硬规则

- 5.6 Sol 是主控，对范围、实现、验证和最终结论负责。
- 禁止调用 Claude 模型、Claude CLI、Claude API 或 Claude 子任务。
- 默认仅使用 Sol；显式调用本 Skill 只授权当前任务最多追加一个 GPT 家族模型。
- 同一时间只允许一个模型写代码，默认顺序执行。
- 未经用户额外批准，不得扩展到第三个模型或并发重复实现。
- 不允许声称使用了未实际可用的模型、努力等级或工具。

## 模型与推理级别

| 模型 | 默认级别 | 定位 |
|---|---|---|
| 5.6 Sol | XHigh | 主控、复杂推理、实现和最终收口 |
| 5.6 Luna | XHigh | 边界明确的批量编码、测试和机械迁移 |
| 5.6 Terra | XHigh | 大范围只读扫描、构建、E2E 和验收 |
| GPT-5.5 | XHigh | 独立架构复核、疑难根因分析或 Sol 不可用时替代 |

只有 5.6 Sol 可以使用 Max。满足以下任一条件才升级：

- 用户明确要求 Max。
- 涉及许可、安全或不可逆数据变更，XHigh 仍无法形成单一结论。
- GPT 家族内部关键证据冲突。
- 同一根因已出现两次有证据的失败修复。

其他 GPT 模型保持 XHigh，不使用 Max。Ultra 不作为默认档。

## 路由

1. 默认：Sol XHigh 单人完成。
2. 批量实现：Sol 定义范围、接口和验收标准，Luna XHigh 实现，Sol 审查验收。
3. 大范围验收：Sol 完成实现，Terra XHigh 只读验证，Sol 处理发现。
4. 架构或根因复核：Sol 负责主线，GPT-5.5 XHigh 只审关键证据或 diff。
5. 争议升级：保持双模型，不增加人数；必要时仅将 Sol 提升到 Max。

## 执行流程

1. Sol 确认目标仓库、脏工作区、不可触碰目录、备份要求和验收标准。
2. Sol 判断单人是否足够；足够时不得启动第二模型。
3. 需要第二模型时，仅传递目标、允许路径、禁止路径、接口、失败证据和验收命令。
4. 执行模型只改获准范围，返回文件清单、关键决策和验证结果。
5. Sol 必须亲自读取 diff、处理发现、运行最终测试并给出结论。
6. 请求的 GPT 模型不可用时继续由 Sol 单人完成，不得偷偷切换 Claude。

## Token 控制

- 不把完整对话和整个仓库交给第二模型。
- 审查优先使用任务简报、`git diff`、测试输出和关键文件。
- 不让 Luna、Terra、GPT-5.5 同时参与同一普通任务。
- 同一问题连续失败两次后升级推理，不增加模型数量。
- 不创建重复报告或重复测试清单。

## 验收

- 运行与变更匹配的单元测试、类型检查、构建和必要的浏览器验收。
- 新增模块目标覆盖率不低于 80%，存量仓库按实际基线报告。
- 检查 `git diff`、秘密信息、用户文件、许可证边界和未处理的运行会话。
- 最终结论明确区分已验证、静态确认、未验证和剩余风险。

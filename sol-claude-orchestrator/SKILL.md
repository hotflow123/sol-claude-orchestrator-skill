# Sol + Claude 双核编排

## 硬规则

- 由 5.6 Sol 对范围、实现、验证和最终结论负责。
- 默认仅使用 Sol；显式调用本 Skill 只授权当前任务最多追加一个模型。
- 同一时间只允许一个模型写代码。默认顺序执行，不并发重复扫描或实现。
- 未经用户额外批准，不得扩展到第三个模型。
- 不因“可能更稳”而重复实现同一功能；独立模型只接收最小任务包。
- 不允许模型声称使用了未实际可用的模型、努力等级或工具。

## 模型与推理级别

| 模型 | 默认级别 | 定位 |
|---|---|---|
| 5.6 Sol | XHigh | 主控、复杂推理、代码所有权、最终收口 |
| claude-fable-5 | XHigh | 与 Sol 同级的高风险独立复核 |
| claude-sonnet-5 | XHigh | 成本较低的常规独立复核 |
| 5.6 Luna | XHigh | 边界明确的批量编码和测试 |
| 5.6 Terra | XHigh | 大范围只读扫描、构建和验收 |
| GPT-5.5 | XHigh | Sol 不可用或 GPT 侧独立复核 |
| Opus 4.8 | XHigh | Fable 不可用时的 Claude 侧替代复核 |

只有 5.6 Sol 和 claude-fable-5 可以使用 Max。满足以下任一条件才升级 Max：

- 用户明确要求 Max。
- 涉及许可、安全、不可逆数据变更，XHigh 证据仍冲突。
- Sol 与 Fable 的关键结论冲突。
- 同一根因已出现两次有证据的失败修复。

不要把 Max 用于普通编码、批量修改、格式化或常规测试。

## 路由

1. 默认：Sol XHigh 单人完成。
2. 批量实现：Sol 定义接口和验收条件，Luna XHigh 实现，Sol 审查并验收。
3. 大范围验收：Sol 完成实现，Terra XHigh 只读验证，Sol 处理发现。
4. 常规独立审查：Sol 完成实现，Sonnet 5 XHigh 只看任务简报、diff 和测试证据。
5. 高风险双核：Sol XHigh 负责主线，Fable 5 XHigh 只在编码前审架构或编码后审 diff，二选一。
6. 争议升级：保持双模型，不增加人数；按升级条件将 Sol 或 Fable 提升到 Max。

Claude 无额度或不可用时，立即执行 `$sol-gpt-orchestrator` 的规则，不反复重试 Claude。

## 执行流程

1. Sol 读取目标代码，确认工作区状态、范围、不可触碰目录和验收标准。
2. Sol 判断是否单人足够；足够时不得启动第二模型。
3. 需要第二模型时，传递最多包含：目标、允许路径、禁止路径、接口、失败证据、验收命令。
4. 写代码前先备份用户要求备份的文件；禁止回退或覆盖用户现有改动。
5. 执行模型只改获准范围，并返回文件清单、关键决策和验证结果。
6. Sol 必须亲自读取 diff，处理问题，运行最终测试并给出结论。

## Token 控制

- 第二模型不得读取完整对话或全仓库，除非范围无法再缩小。
- 审查模型优先读取 `git diff`、失败日志和相关文件，不重新做完整探索。
- 每个阶段只保留一个明确产物；不要生成重复报告。
- 连续两次遇到同一阻塞时停止重复尝试，升级推理而不是增加模型数量。
- 普通任务不得同时调用 Fable、Sonnet、Luna、Terra。

## 验收

- 运行与变更匹配的单元测试、类型检查、构建和必要的浏览器验收。
- 新增模块的目标覆盖率不低于 80%，存量仓库按实际基线报告。
- 检查 `git diff`、秘密信息、用户文件、许可证边界和未处理的运行会话。
- 最终结论必须区分：已验证、仅静态确认、未验证和剩余风险。
